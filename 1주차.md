
### 1. 학습 과정과 현재 자신이 생각하는 역량은 무엇인가요? (1000자 이내)
제가 생각하는 학습 과정의 핵심은 ‘지금 당장 필요하지 않더라도 다양한 기술과 사례를 경험해보는 것’입니다. 개발 환경은 계속 변화하고 새로운 기술도 빠르게 등장하기 때문에, 저는 폭넓은 경험을 통해 상황에 맞는 최적의 선택을 할 수 있는 개발자가 되고자 노력해왔습니다. 그래서 새로운 기술을 배우면 직접 실험해보고 프로젝트에 적용해보며, 제 도메인에 맞게 어떻게 사용할 수 있을지 항상 고민합니다.

이러한 저만의 학습 방식이 가장 잘 드러났던 경험은 첫 회사에서의 프로젝트입니다. 당시 저는 **사수가 없는 환경에서 백엔드 시스템을 처음부터 끝까지 독립적으로 개발해야 했습니다.** 설계부터 DB 모델링, 배포까지 모든 결정을 스스로 내려야 했고, 참고할 사람이 없었습니다. 그래서 저는 한 가지 원칙을 세웠습니다.

**“대기업의 검증된 기술 사례를 내 환경과 비교·분석하여, 적용 가능한 부분만 선별하자.”**

이 원칙을 바탕으로 여러 대기업의 기술 컨퍼런스 발표와 문서를 찾아 분석했습니다.

“이 회사는 어떤 이유로 이 아키텍처를 선택했지?”,

“내 프로젝트 규모에서도 이 방식이 유효할까?”

와 같은 질문을 스스로 던지며, 그대로 따라가는 것이 아니라 ‘제 환경에 맞게 재해석하는 방식’으로 학습했습니다.

이렇게 쌓은 경험의 조각들은 실제로 **코인거래소 스테이킹 서비스에서 이자 지급 기능을 개발하며 Spring Batch를 적용할 때** 큰 도움이 되었습니다. 초기 실험을 통해 배치 처리 방식과 트랜잭션 관리 방법을 익혀두었기 때문에, 실제 구현 과정에서도 문제 없이 빠르게 적용할 수 있었습니다.

그 결과, **스테이킹 서비스 신청부터 지급·해제까지 전 과정에서 이슈 티켓 0건을 달성했고, 서비스는 가상 환경에서 인기 상품으로 선정되는 성과를 얻었습니다.**

### 2. 선택한 직군을 목표로 하게 된 이유는 무엇인가요? (1000자 이내)
어렸을 때부터 “사람이 반복적으로 하는 일을 자동화하면 더 가치 있는 일에 집중할 수 있지 않을까?”라는 생각을 자주 했습니다. 일상에서 불필요하게 낭비되는 시간과 수고를 줄이는 데 흥미를 느꼈고, 스마트폰과 인터넷이 일상이 된 세상에서 **가장 큰 변화를 만들 수 있는 직업이 개발자**라고 생각했습니다.

개발을 처음 접했을 때 작은 기능 하나로도 불편함을 줄일 수 있다는 사실을 배우는 과정 자체가 흥미로웠고, **이 경험이 개발자가 되고 싶다는 확신으로 이어졌습니다.**

이러한 확신을 바탕으로 컴퓨터공학과에 진학해 자료구조, 네트워크, 운영체제 등 CS 지식을 체계적으로 학습했습니다. 학부 과정에서 배운 이론이 실제 서비스에서 어떻게 활용되는지 궁금했고, 이를 검증하기 위해 여러 사이드 프로젝트를 진행했습니다. 또한 실무 역량을 쌓기 위해 온라인 강의, 기술 블로그 운영 등을 꾸준히 병행했습니다.

회사에 입사한 이후에는 실제 사용자가 있는 서비스를 개발하고 운영하며 개발의 의미를 더 깊게 이해하게 되었습니다. 특히 서비스의 병목지점을 개선하거나 반복적인 업무를 자동화해 팀의 효율을 높였을 때 큰 보람을 느꼈습니다. 단순히 기능을 만드는 것이 아니라, **문제를 정확히 정의하고 해결하는 과정 자체가 개발의 본질**이라는 것을 깨달았습니다.

최근에는 CS 이슈가 가장 많았던 채팅 기능을 개선해 배포하면서 고객센터 문의가 눈에 띄게 줄어든 경험이 있습니다. 작지만 사용자의 불편을 직접 해결한 개선이 큰 효과로 이어지는 것을 보며, 개발자를 선택하길 잘했다는 확신을 다시 한 번 느꼈습니다.

저는 “개발은 문제 해결”이라고 생각합니다. 앞으로도 더 많은 문제를 발견하고, 기술로 해결하는 개발자가 되고 싶습니다. 누군가의 불편함을 줄여주고 시간을 절약해주는 시스템을 만드는 것. 이것이 제가 개발자를 선택한 이유이자, 지금도 개발자로 성장하고 싶은 가장 큰 동기입니다.

### 3. 마음이 끌리는 무언가에 몰입하여 일정 수준의 결과물을 만들어 낸 경험이 있나요? (1000자 이내)
평소에 “언젠가는 세계적인 오픈소스에 기여해보고 싶다”는 목표가 있었습니다. 그러나 방대한 코드베이스를 어떻게 이해해야 하는지, 어디서부터 시작해야 하는지 막막함이 더 컸기 때문에 쉽게 도전하지 못하고 있었습니다. 그러던 중 AI 코드 분석 도구(ChatGPT 등)를 활용하면서 상황이 달라졌습니다. 복잡한 코드 구조와 의도를 빠르게 파악할 수 있었고, “이제는 실행해볼 수 있겠다”는 확신이 생겼습니다.

그때 제가 선택한 프로젝트가 **Spring Kafka**였습니다. 실제 업무에서 Kafka를 사용하면서, 특정 설정 객체의 `toString()` 메서드가 문자열을 `+` 연산으로 계속 연결하는 방식으로 작성되어 있어 불필요한 객체 생성과 성능 저하를 유발할 수 있다는 점을 발견했습니다. 이 문제를 개선하고 싶다는 생각이 들어 직접 PR을 작성하기로 결정했습니다.

저는 코드를 `StringBuilder` 기반으로 리팩터링하여 메모리 할당을 줄이고, 문자열 연산 비용을 최소화했습니다. 또한 내부 중복 코드를 정리해 유지보수성을 높였으며, 출력 포맷은 기존과 동일하게 유지해 사용자 영향 없이 성능을 개선할 수 있도록 했습니다.

하지만 이 과정이 순탄한 것은 아니었습니다. 리뷰어의 기준을 충족하기 위해 여러 차례 커밋을 수정해야 했고, “왜 이 변경이 필요한가?”를 명확한 근거로 설명해야 했습니다. 특히 리뷰어가 해외에 있어 **새벽 시간에 리뷰 알림이 도착**했는데, 저는 피드백을 놓치지 않기 위해 알림이 올 때마다 바로 대응하며 의견을 조율했습니다. 단순히 코드를 수정하는 것이 아니라, **설득과 소통**을 통해 리뷰어의 신뢰를 얻는 과정이 필요했습니다.

AI는 이 과정에서 큰 도움이 되었습니다. 코드 변경의 영향 범위를 빠르게 분석해줬고, 리뷰어가 우려할 지점을 미리 확인하며 대응할 수 있었기 때문에 리뷰 과정이 훨씬 효율적이었습니다. 이런 반복적인 수정과 논리적 설명 끝에, 결국 PR은 승인되어 **Spring Kafka 메인 저장소에 머지되는 성과**를 얻을 수 있었습니다.

이 경험을 통해 얻은 깨달음은 명확했습니다.

- **생각만 하면 아무것도 바뀌지 않는다. 실행해야 결과가 생긴다.**
- **AI는 개발자를 대체하는 도구가 아니라, 실행력을 극대화해주는 파트너이다.**
- **세계적인 프로젝트에 기여하는 사람도 특별한 사람이 아니라, 도전하고 끝까지 밀어붙인 사람이다.**

무엇보다 이 경험은 “얼마나 한 가지 일에 깊게 몰입할 수 있는가”,

그리고 **의사소통의 어려움 속에서도 끝까지 해결하려는 집요함**을 스스로 입증한 경험이었습니다.

이제 저는 새로운 기술을 사용할 때 단순한 ‘사용자’가 아니라, **문제를 발견하면 직접 개선하는 개발자**가 되기 위해 계속 도전하고 있습니다. 이 작은 시작을 기반으로 앞으로도 꾸준히 세계적인 오픈소스에 기여하고, 기술 생태계에 영향을 주는 개발자로 성장하고 싶습니다.

### 4. 우리 회사에 지원하려는 이유는 무엇인가요? (500자 이내)
"정산플랫폼팀이 매일 수백억 원의 데이터를 Spring Batch로 처리한다는 점에서 제 경험과 강한 시너지를 낼 수 있다고 확신합니다.

저는 Spring Batch를 활용하여 대용량 데이터 처리 중 발생하는 업데이트 누락 문제를 커스텀 PagingReader를 구현해 해결한 경험이 있으며, 이 과정에서 배치 성능 최적화의 중요성을 깊이 체감했습니다. 특히 Spring Batch 오픈소스에 직접 기여한 PR #4931에서 DefaultFieldSet의 indexOf() 성능을 O(n)에서 O(1)로 개선하여, CSV 파일 처리 성능을 향상시켰습니다. 이는 정산 데이터처럼 컬럼이 많은 대용량 파일 처리에 직접적인 도움이 되는 개선사항입니다.

정산플랫폼팀이 추구하는 ""복잡함을 단순함으로 바꾸는"" 철학은 제가 코드 레벨에서 추구해온 가치관과 일치합니다. 정산이라는 미션 크리티컬한 도메인에서 신뢰성 있는 시스템을 구축하고, 3000개 이상의 테스트 코드로 비즈니스를 뒷받침하는 문화 속에서 성장하고 싶습니다. 제 Spring Batch 최적화 경험과 문제 해결 능력으로 팀의 ""정산을 쉽게 만드는 플랫폼"" 미션에 기여하고 싶습니다."
