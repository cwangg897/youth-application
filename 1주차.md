
### 1. 학습 과정과 현재 자신이 생각하는 역량은 무엇인가요? (1000자 이내)
"제가 생각하는 학습 과정의 핵심은 ‘당장 사용하지 않더라도 다양한 경험을 해보는 것’이라고 믿습니다. 기술은 빠르게 바뀌고, 정답도 없기 때문에 하나의 기술에만 머물러 있는 것은 오히려 위험하다고 생각합니다. 저는 새로운 기술을 배우면 프로젝트에서 반드시 써보려고 하고, 필요하지 않더라도 실험해보며 “경험치”를 쌓는 방식을 선호합니다. 이 방식 덕분에 문제를 마주했을 때 선택지가 넓어지고, 여러 상황에 유연하게 대응할 수 있게 되었습니다.

이러한 학습 방식이 가장 강하게 드러났던 경험은 첫 회사에서의 프로젝트입니다. 당시 저는 사수가 없는 환경에서 혼자 백엔드 시스템을 처음부터 끝까지 개발해야 했습니다. 설계, DB 모델링, 배포까지 스스로 결정해야 했고, 의논할 사람조차 없었습니다. 답이 없는 상황에서 저는 ‘최대한 많이 보고 배우자’라는 방향을 잡았습니다.

그때부터 저는 대기업 기술 컨퍼런스 영상을 찾아보기 시작했습니다. 라인 기술 블로그, 우아한테크세미나, 카카오 기술 블로그 등 여러 기업의 발표와 문서를 보며, “이 회사는 트래픽을 어떻게 처리하지?”, “이 아키텍처는 왜 선택했지?” 와 같은 질문을 스스로 던졌습니다. 실무에서 바로 쓰이지 않는 내용도 많았지만, 저는 ‘이해가 안 되더라도 일단 부딪히는 것’이 학습이라고 생각했습니다. 그렇게 쌓인 조각들이 나중에 실제 시스템 장애 대응, 구조 개선, 성능 최적화 문제를 해결할 때 큰 자산이 되었습니다.

이 경험을 통해 제가 얻은 역량은 다음과 같습니다.

1. 문제가 생기면 주도적으로 해결하는 태도
    
    답을 알려주는 사람이 없어도, 찾아보고, 검증하고, 결정할 수 있는 역량입니다.
    
2. 기술 선택의 근거를 설명할 수 있는 사고력
    
    “왜 이 기술을 쓰는가?”에 대해 설명하지 못하면 선택하지 않습니다.
    
3. 읽고, 이해하고, 구조화하는 능력
    
    다른 회사의 사례를 내 문제 해결에 적용하는 역량입니다.
    

저는 빠르게 배우는 것이 중요한 것이 아니라, 스스로 배울 수 있는 사람이 되는 것이 중요하다고 생각합니다. 다양한 경험을 통해 미래의 문제를 해결할 수 있는 힘을 길러왔고, 앞으로도 **실행하고 실험하는 개발자**로 성장하고자 합니다."

### 2. 선택한 직군을 목표로 하게 된 이유는 무엇인가요? (1000자 이내)
"어렸을 때부터 “사람이 반복적으로 하는 일을 자동화하면 더 가치 있는 일에 집중할 수 있지 않을까?”라는 생각을 자주 했습니다. 일상에서 불필요하게 낭비되는 시간과 수고를 줄이는 것에 흥미를 느꼈고, 이를 실현할 수 있는 직업이 개발자라고 생각했습니다. 단순한 호기심에서 출발했지만, 제가 만든 프로그램이 누군가의 시간을 줄여주고 업무를 더 효율적으로 만들어주는 경험을 하면서 점점 확신으로 바뀌었습니다.

이런 확신으로 컴퓨터공학과에 진학해 자료구조, 네트워크, 운영체제 등 CS 기반 지식을 체계적으로 학습했습니다. 학부 과정에서 배운 내용이 실제 서비스에서 어떻게 사용되는지 궁금했고, 이를 검증하기 위해 여러 사이드 프로젝트를 진행했습니다. 또한 실무에 필요한 역량을 갖추기 위해 온라인 강의와 기술 블로그 운영 등을 꾸준히 병행했습니다.

회사에 입사한 이후로는 실제 사용자가 있는 서비스를 개발하고 운영하는 경험을 했습니다. 특히 서비스의 병목지점을 개선하거나, 반복적인 업무를 시스템화하여 팀의 효율을 높였을 때 가장 큰 보람을 느꼈습니다. 단순히 기능을 개발하는 것이 아니라, 문제를 정의하고 해결하는 과정에서 제 일의 의미를 찾는다는 것을 깨달았습니다.

저는 “개발은 문제 해결”이라고 생각합니다. 앞으로도 더 많은 문제를 발견하고, 기술로 해결하는 개발자가 되고 싶습니다. 누군가에게 불편함을 줄여주고, 사용자의 시간을 절약하는 시스템을 만드는 것. 이것이 제가 개발자를 선택한 이유이며, 지금도 개발자로 성장하고 싶은 가장 큰 동기입니다."

### 3. 마음이 끌리는 무언가에 몰입하여 일정 수준의 결과물을 만들어 낸 경험이 있나요? (1000자 이내)
"평소에 “언젠가는 세계적인 오픈소스에 기여해보고 싶다”는 생각을 해왔습니다. 하지만 방대한 코드베이스를 이해하는 것 자체가 부담이었고, 어디서부터 시작해야 할지 몰라 실행하지 못하고 있었습니다. 그러던 중 AI 코드 분석 도구(ChatGPT 등)를 활용하면서 상황이 크게 달라졌습니다. 복잡한 코드도 AI에게 구조와 의도를 물어보며 빠르게 이해할 수 있었고, “이제는 나도 할 수 있겠다”는 확신이 생겼습니다.

그 과정에서 제가 선택한 프로젝트는 Spring Kafka였습니다. 실제 업무에서 Kafka를 사용하면서, 특정 설정 객체의 `toString()` 메서드가 문자열을 `+` 연산으로 계속 연결하는 방식으로 작성되어 있어 불필요한 객체가 생성되고 성능 저하를 야기할 수 있다는 문제점을 발견했습니다. 저는 이 부분을 개선하기 위한 PR을 직접 작성했으며, 개선 내용은 다음과 같습니다.

코드를 `StringBuilder` 기반으로 리팩터링하여 메모리 할당과 문자열 연산 비용을 줄이고, 내부적으로 중복되는 코드를 정리하여 유지보수성을 높였습니다. 변경 후에도 출력 포맷은 동일하게 유지되도록 처리하여 기존 사용자에게 영향이 없도록 했습니다.

처음에는 리뷰어의 기준에 맞추기 위해 커밋을 여러 번 수정해야 했고, 왜 이 변경이 필요한지 논리적으로 설명해야 했습니다. AI가 코드의 영향 범위를 빠르게 분석해 주었기 때문에, 그 덕분에 리뷰 과정을 효과적으로 진행할 수 있었습니다. 결국 PR은 승인되어 Spring Kafka 메인 저장소에 머지되는 경험을 할 수 있었습니다.

이 경험을 통해 깨달은 것은 다음과 같습니다.

- 생각만 하면 아무것도 달라지지 않는다. 실행해야 바뀐다.
- AI는 개발자의 역량을 대체하는 것이 아니라, 실행력을 높여주는 도구다.
- “세계적인 프로젝트에 기여하는 사람”도 특별한 사람이 아니라, 실행한 사람일 뿐이다.

이 경험은 저에게 큰 자신감을 주었습니다. 이제는 새로운 기술을 사용할 때 단순 사용자 뿐만 아니라 개선하는 개발자가 되기 위해 계속 도전하고 있습니다."

### 4. 우리 회사에 지원하려는 이유는 무엇인가요? (500자 이내)
"정산플랫폼팀이 매일 수백억 원의 데이터를 Spring Batch로 처리한다는 점에서 제 경험과 강한 시너지를 낼 수 있다고 확신합니다.

저는 Spring Batch를 활용하여 대용량 데이터 처리 중 발생하는 업데이트 누락 문제를 커스텀 PagingReader를 구현해 해결한 경험이 있으며, 이 과정에서 배치 성능 최적화의 중요성을 깊이 체감했습니다. 특히 Spring Batch 오픈소스에 직접 기여한 PR #4931에서 DefaultFieldSet의 indexOf() 성능을 O(n)에서 O(1)로 개선하여, CSV 파일 처리 성능을 향상시켰습니다. 이는 정산 데이터처럼 컬럼이 많은 대용량 파일 처리에 직접적인 도움이 되는 개선사항입니다.

정산플랫폼팀이 추구하는 ""복잡함을 단순함으로 바꾸는"" 철학은 제가 코드 레벨에서 추구해온 가치관과 일치합니다. 정산이라는 미션 크리티컬한 도메인에서 신뢰성 있는 시스템을 구축하고, 3000개 이상의 테스트 코드로 비즈니스를 뒷받침하는 문화 속에서 성장하고 싶습니다. 제 Spring Batch 최적화 경험과 문제 해결 능력으로 팀의 ""정산을 쉽게 만드는 플랫폼"" 미션에 기여하고 싶습니다."
